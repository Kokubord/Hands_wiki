## ALERTA MÁXIMO — o dev **não** pode gravar na Neon / produção (2026-04-23)

**Isto não é um objectivo — é uma proibição.** A secção abaixo descreve um **incidente** em que o `next dev` **ligou por engano** à Neon (dados reais) apesar dos `.env*` locais — comportamento **ilegítimo** que o repositório passou a impedir tecnicamente e a proibir por política.

### O que falhou (para nunca repetir)

- **Sintoma:** `http://localhost:4000` mostrava dados de **Clinica Inspirar** (organograma, etc.) enquanto `docker exec … psql` na base **local** mostrava **seed Demo** (Contratante Demo, Unidade Centro/Sul/Leste).
- **Causa raiz:** o processo **`next-server`** na porta 4000 tinha **`DATABASE_URL` herdada do ambiente** (shell / Cursor / tarefa) apontando para **Neon** (`*.neon.tech`). O Next.js **não sobrepõe** variáveis já definidas no processo — os ficheiros `.env.local` com `localhost` eram **ignorados** para efeitos de ligação Prisma.
- **Falso conforto:** o script `predev` (`assert-dev-database-is-local.ts`) corria noutro processo Node com `dotenv` e `override: true`, validava **só o que estava nos ficheiros**, e saía com sucesso — **sem corrigir** o ambiente que o filho `next dev` ia herdar.
- **Correcção no repo:** `npm run dev` / `dev:free` / `run-dev-wsl.sh` passam a arrancar o Next via `scripts/next-dev-local.sh`, que faz **`env -u DATABASE_URL -u DATABASE_URL_UNPOOLED`** antes do `next dev` (excepto se `CLINICAGESTOR_DEV_ALLOW_REMOTE_DATABASE=true`). O assert passou a **avisar** quando o shell tinha host remoto mas os `.env*` estão locais. Ver também `README.md` e `.env.example`.

### Política inegociável (agentes LLM e humanos)

### A ÚNICA ROTA DE TRANSPORTE PARA PRODUÇÃO É: **PUSH PARA O GITHUB** E **DEPLOY DE PRODUÇÃO** (PIPELINE / VERCEL OU EQUIVALENTE).

### **JAMAIS** «MUDAR A ROTA» DA BASE DE DADOS DO **AMBIENTE DE DEV LOCAL** PARA APONTAR PARA **PRODUÇÃO**, **STAGING**, **NEON REMOTO** OU QUALQUER POSTGRES PARTILHADO COMO ATALHO OPERACIONAL.

- **Proibido** a agentes: alterar `.env`, `.env.local`, `.env.development.local`, `schema.prisma`, scripts de arranque ou documentação com o **efeito** de desviar o fluxo normal **dev local → Docker/localhost** sem ordem **explícita e por escrito** do dono do produto.
- **Proibido** «resolver» bugs de dados a apontar o laptop para a base remota em silêncio. Se alguém precisar de depuração contra uma base remota, isso é **excepção** documentada e consciente (`CLINICAGESTOR_DEV_ALLOW_REMOTE_DATABASE=true` + revisão humana), **nunca** o caminho por omissão.
- **Incumprimento** = risco de corrupção de dados reais, violação de confiança e auditoria; trate-se como **incidente de segurança operacional**, não como «conveniência de dev».

*(A fórmula «sob pena de morte» no chat do utilizador reflecte a gravidade percebida; no repositório formalizamos como **regra absoluta / zero tolerância**.)*

## Convenções em vigor

- **Feature flags por clínica** para integrações externas: ex. `legalSignatureLacunaEnabled`, `legalSignatureVidaasEnabled` em `Clinic`.
- **Provider registry central:** `lib/clinical/legal-signature-providers.ts` lista sempre todos os fornecedores (mesmo «em preparação») para transparência do médico.
- **Audit events** separados: `CONSULTATION_EPISODE_SIGNED` (encerramento no sistema) vs `CONSULTATION_EPISODE_LEGAL_SIGNATURE_COMPLETED` (assinatura ICP-Brasil) — não confundir.
- **UI congelada:** blocos marcados «layout oficial — não alterar sem…» ou UIs já aprovadas por produto NÃO são alteradas sem consentimento explícito. Ver `.cursor/rules/frozen-ui-approval.mdc`.
- **Confirmação antes de eliminar dados:** botões ou formulários que apaguem ou cancelem dados persistentes devem exigir confirmação explícita do utilizador antes da submissão (padrão actual: `DestructiveConfirmForm` + diálogo nativo; ver «Decisões frescas», 2026-04-22). Novos fluxos destrutivos devem seguir o mesmo critério.
- **Commit convention:** Conventional Commits + dois tags especiais — `DECISION(scope)` para decisão material (deve referenciar ADR), `HANDOVER(scope)` para actualização deste ficheiro. Ver `AGENTS.md` na raiz do repositório.
- **Idioma:** docs em português europeu (matching do estilo existente nos ADRs).
